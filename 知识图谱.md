# 知识图谱

## 概念

- **定义：**结构化语义知识库，通过图结构（节点 - 实体，边 - 关系）描述现实世界实体、概念及关联。
- **核心：**“实体 - 关系 - 实体”三元组（如“苹果公司 - 生产 - iPhone”）
- **本质：**将分散信息转化为==可推理==的网状知识结构，为AI提供认知框架。

## 构建流程

![image-20260412190420745](/Users/des1re/Pictures/笔记图片/image-20260412190420745.png)

1. 数据收集与准备；
   整合结构化、半结构化、非结构化数据。
2. 明确领域与目标；
   确定实体、关系及应用场景。
3. 实体识别与关系抽取；
   NLP技术提取实体及关系。
4. 知识图谱构建；
   图数据库存储形成图结构。
5. 优化与维护。
   新增、修正、更新数据源。

### 数据收集与准备

![image-20260412190935445](/Users/des1re/Pictures/笔记图片/image-20260412190935445.png)

本体保证后面的内容都是统一、规范、有条理的，避免不同实体之间的突然关系链接，例如药品和症状出现父子关系。

### 知识抽取

![image-20260412191123660](/Users/des1re/Pictures/笔记图片/image-20260412191123660.png)

给定一篇文章，需要从其中抽取相关的医学实体，基于==NLP==或者==大模型==完成三元组的提取。

### 知识融合

![image-20260412191258442](/Users/des1re/Pictures/笔记图片/image-20260412191258442.png)

对于同一个内容有不同说法（提及），需要融合为统一实体。

### 知识存储

![image-20260412191544979](/Users/des1re/Pictures/笔记图片/image-20260412191544979.png)

将不同数据（知识三元组）存储在数据库中，这个数据库通常是专门处理图数据的，例如neo4j等。

## 应用

在有了知识图谱后，如何进行下游任务？问答、可视化、推荐，以问答为例。

### 问答

基于知识图谱等问答引擎，以医疗问答为例。对于打喷嚏、四肢酸痛的症状描述：

- 首先在知识图谱中进行推理，找到==打喷嚏、四肢酸痛==两个节点，有了节点后，沿着关系网络进行==多跳查询==，发现两个症状都指向了==感冒==这一节点；
- 从感冒这一节点中找到==推荐用药==这一关系，从而查询对应==药品==节点；
- 组织语言回复：症状描述 -> 感冒 -> 推荐用药 -> 药品。

# GraphRAG

对于传统RAG，只适用于局部问题（明确的检索任务），不适用于==查询聚焦摘要（QFS）==任务，无法结合跨片段知识进行回答，即无法处理全局性问题。

## 核心思想

构建了一个基于图的索引，利用LLM将实体信息串起来，形成一个知识图谱。此外，利用==社区检测==算法，将图分成一个个主题紧密相关的模块，就像将大图书馆按学科分类。

预先为每个模块生成摘要报告，当用户提问时，将所有相关模块摘要进行回答，得到一堆部分答案，再将部分答案汇总，得到全局答案。
